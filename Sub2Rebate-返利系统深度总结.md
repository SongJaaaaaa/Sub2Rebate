# Sub2Rebate 独立分销返利系统：需求定稿与可实施方案

> 文档目的：把前期研讨内容沉淀成可实施的产品方案。  
> 当前阶段：需求确认与方案定稿，不进入代码实现。  
> 核心目标：在不改 Sub2API 核心代码的前提下，独立实现邀请关系、里程碑返利、多级分发、支付宝充值提现、防躺平、风控与财务审计。

---

## 1. 项目结论

本项目建议建设一个独立系统 **Sub2Rebate**，与现有 Sub2API 解耦运行。

Sub2API 继续负责：

- AI API 中转。
- 用户账号与额度消费。
- 原有管理员后台与基础用户体系。

Sub2Rebate 独立负责：

- 推广邀请关系。
- 树状多级分销链路。
- 新人累计充值里程碑奖励。
- 里程碑结束后的多级返利分发。
- 独立返利余额。
- 支付宝充值与提现。
- 防躺平贡献系数。
- 风控、审计、对账、配置管理。

最终推荐架构：

```text
用户 / 推广员
    |
    v
Sub2Rebate 前台
    |-- 推广链接 / 邀请关系
    |-- 支付宝充值
    |-- 返利余额 / 提现
    |
    v
Sub2Rebate 服务端
    |-- 邀请链路引擎
    |-- 里程碑奖励引擎
    |-- 多级返利引擎
    |-- 贡献系数引擎
    |-- 风控与审计
    |
    v
PostgreSQL
    |-- Sub2API 原有表
    |-- Sub2Rebate 独立业务表
    |
    v
Sub2API
    |-- 额度发放
    |-- API 消费
```

---

## 2. 为什么不直接改 Sub2API

前期讨论过三种路径：

| 方案 | 结论 | 原因 |
|---|---|---|
| 直接改 Sub2API 源码 | 不推荐 | 上游更新会覆盖或冲突，后续维护成本高 |
| Fork 后维护自定义分支 | 可行但不优先 | 适合小改动，不适合独立提现、多级分销、复杂活动 |
| 独立 Sub2Rebate 系统 | 推荐 | 升级不受影响，业务可自由扩展，财务独立 |

最终选择独立系统的原因：

- Sub2API 原生返利只覆盖部分场景，尤其管理员手动加额度无法触发返利。
- 本项目需求已经不只是“补一个返利触发点”，而是完整分销系统。
- 后续会涉及支付宝支付、提现审核、风控、活动配置、贡献系数，这些都不适合强塞进 Sub2API。

## 3. 已确定需求

### 3.1 业务目标

已确定：

- 用于大规模推广，不是小范围内部拼车。
- 返利系统独立运行。
- 提现放在新系统，不依赖 Sub2API 原生余额。
- 需要支持树状裂变，一个人可以发展多个下级。
- 需要支持多级返利，但直接拉人的人必须拿最多。
- 需要防止上级坐享其成，即防躺平机制。
- 所有规则尽量后台可配置，减少改代码。
- 所有配置项必须有 Tips 说明。
- 所有返利和提现必须有完整流水，方便对账。

### 3.2 技术方向

已确定：

- Sub2Rebate 独立部署。
- 推荐与 Sub2API 共用 PostgreSQL。
- 优先使用数据库监听或数据读取方式识别充值、额度变化。
- 不优先使用 Admin API 轮询作为核心方案。
- 不直接把返利余额混入 Sub2API balance。

### 3.3 财务方向

已确定：

- Sub2API 额度用于 API 消费。
- Sub2Rebate 返利余额用于提现。
- 两套余额独立。
- 支付宝充值、提现后续在 Sub2Rebate 完成。
- 提现需要审核、流水、风控。

## 6. 邀请关系设计

### 6.1 核心原则

Sub2Rebate 必须维护自己的邀请关系，不完全依赖 Sub2API 原生邀请展示。

原因：

- Sub2API 原生邀请码只能说明直接邀请关系。
- 多级分销需要完整上级链路。
- 未来要支持树状裂变、统计、风控和多级计算。

### 6.2 邀请链路结构

推荐使用 Materialized Path。

示例：

```text
A
├── B
│   ├── D
│   └── E
└── C
    └── F
```

路径示例：

| 用户 | parent_id | path | depth |
|---|---|---|---:|
| A | null | A | 0 |
| B | A | A.B | 1 |
| C | A | A.C | 1 |
| D | B | A.B.D | 2 |
| E | B | A.B.E | 2 |
| F | C | A.C.F | 2 |

当 D 充值时，只沿 D 的上级链路计算：

```text
D -> B -> A
```

当 F 充值时，只沿 F 的上级链路计算：

```text
F -> C -> A
```

### 6.3 绑定规则

建议规则：

- 用户首次注册或首次绑定邀请码时确定上级。
- 绑定后默认不可自行更换上级。
- 管理员可手动修正，但必须记录操作日志。
- 禁止自己邀请自己。
- 禁止形成循环关系。
- 同一用户只能有一个直接上级。

## 7. 返利规则总览

返利分为两个阶段。

```text
新人被邀请
    |
    v
累计充值是否还在里程碑奖励次数内？
    |
    |-- 是 -> 只奖励直接上级
    |
    |-- 否 -> 进入正常多级分发
```

阶段一：新人累计充值里程碑奖励。

- 强激励直接拉人的人。
- 只给直接上级。
- 第 2 级以上不参与。

阶段二：正常阶级多级分发。

- 在里程碑次数用完后启用。
- 沿邀请链路向上分发。
- 第 1 级最多，越往上越少。
- 可叠加贡献系数。
- 最终归一化控制预算。

## 8. 返利池规则

### 8.1 默认规则

默认：

```text
用户充值 100 元
平台拿出 15 元作为返利池
```

等价于：

```text
返利池比例 = 15%
```

### 8.2 必须可配置

后台配置：

| 配置项 | 默认值 | 说明 |
|---|---:|---|
| 返利池计算方式 | 按比例 | 按充值金额计算返利池 |
| 返利池比例 | 15% | 充值 100 元返利池为 15 元 |
| 里程碑金额 | 100 元 | 新人累计充值达到多少触发一次奖励 |
| 里程碑奖励金额 | 15 元 | 每次里程碑给直接上级多少钱 |

### 8.3 活动调整

需要支持活动临时调整：

示例：

- 平时：100 返 15。
- 活动：100 返 20。
- 活动期结束后自动恢复。

活动配置需要包含：

- 活动名称。
- 生效时间。
- 结束时间。
- 返利池比例。
- 是否影响里程碑奖励。
- 是否影响正常多级分发。
- 备注。

## 9. 阶段一：里程碑奖励

### 9.1 规则定义

新人累计充值达到一个里程碑后，直接上级获得一次固定奖励。

默认：

```text
里程碑金额 = 100 元
每次奖励 = 15 元
奖励次数 = 3 次
```

解释：

```text
新人累计充值达到 100 元 -> 直接上级拿 15 元
新人累计充值达到 200 元 -> 直接上级再拿 15 元
新人累计充值达到 300 元 -> 直接上级再拿 15 元
之后该新人再充值 -> 进入正常多级分发
```

### 9.2 关键点

必须遵守：

- 按新人累计充值计算，不是单笔充值。
- 奖励只给直接上级。
- 更上级不参与。
- 奖励次数可配置。
- 里程碑金额可配置。
- 每次奖励金额可配置。
- 必须防重复发放。

### 9.3 示例

配置：

```text
里程碑金额：100
里程碑奖励：15
奖励次数：3
```

新人 D 由 C 邀请。

| D 的充值行为 | 累计充值 | 触发结果 |
|---|---:|---|
| 充值 60 | 60 | 不触发 |
| 再充值 50 | 110 | 触发第 1 次，C +15 |
| 再充值 100 | 210 | 触发第 2 次，C +15 |
| 再充值 100 | 310 | 触发第 3 次，C +15 |
| 再充值 100 | 410 | 进入正常多级分发 |

### 9.4 边界场景

#### 单笔跨多个里程碑

如果 D 首次直接充值 300：

```text
累计从 0 到 300
跨过 100、200、300 三个里程碑
如果奖励次数 = 3，则一次性给 C 发 3 次奖励，共 45
```

#### 累计超过但次数已用完

如果奖励次数 = 3，D 累计充值达到 400：

```text
前 300 对应 3 次里程碑奖励
超过部分进入正常多级分发
```

该“超过部分如何进入多级分发”需要实现时明确口径：

推荐口径：

- 以实际充值事件为单位计算。
- 当本次充值导致里程碑次数用完后，本次充值中超过最后里程碑的金额可进入正常多级分发。
- 为降低复杂度，MVP 可以从下一笔充值开始进入多级分发。

建议 MVP 先采用简单口径：

> 里程碑次数用完后的下一笔充值，开始正常多级分发。

## 10. 阶段二：正常多级分发

### 10.1 触发条件

某个新人对其直接上级的里程碑奖励次数已经用完后，该新人后续充值进入正常多级分发。

### 10.2 推荐默认算法：固定比例模式

MVP 推荐先使用固定比例模式，因为可解释性强、后台配置简单。

默认比例：

| 层级 | 比例 | 100 元充值、15 元返利池时金额 |
|---|---:|---:|
| 第 1 级：直接上级 | 60% | 9.00 |
| 第 2 级 | 20% | 3.00 |
| 第 3 级 | 10% | 1.50 |
| 第 4 级 | 6% | 0.90 |
| 第 5 级 | 4% | 0.60 |
| 合计 | 100% | 15.00 |

这个配置符合当前共识：

- 直接劳动者拿最多。
- 上级能拿，但逐级减少。
- 返利池全部分配。
- 规则简单，用户容易理解。

### 10.3 可选算法：衰减系数模式

后续可支持衰减系数。

示例：

```text
第 1 级基础权重 = 1
衰减系数 = 0.4

第 1 级权重 = 1
第 2 级权重 = 0.4
第 3 级权重 = 0.16
第 4 级权重 = 0.064
第 5 级权重 = 0.0256

再把权重归一化到 100%
```

优点：

- 层级可动态扩展。
- 不需要手动配置每一级。

缺点：

- 普通用户理解成本高。
- 后台 Tips 必须写清楚。

建议：

- MVP 先做固定比例。
- 第二阶段增加衰减系数模式。

### 10.4 动态层级

系统必须按真实邀请链路计算。

例如：

```text
A -> B -> C -> D 充值
```

D 的上级链路：

```text
第 1 级：C
第 2 级：B
第 3 级：A
```

如果后台配置最多 5 级，但实际只有 3 级：

- 只发给 C、B、A。
- 第 4、5 级无人，不发。

这里有一个产品口径需要确定：

| 场景 | 方案 A | 方案 B |
|---|---|---|
| 实际层级少于配置层级 | 未发出的比例归平台 | 对已有上级重新归一化分完 |

建议选择 **方案 B：对已有上级重新归一化分完**。

原因：

- 用户之前强调“15 元全部分配，不需要考虑剩多少归平台”。
- 更符合返利池全部用于推广的目标。

示例：

配置比例：

```text
60 / 20 / 10 / 6 / 4
```

实际只有 3 级：

```text
原权重：60 + 20 + 10 = 90
归一化后：
第 1 级 = 60 / 90 = 66.67%
第 2 级 = 20 / 90 = 22.22%
第 3 级 = 10 / 90 = 11.11%
```

15 元返利池：

```text
第 1 级 = 10.00
第 2 级 = 3.33
第 3 级 = 1.67
```

## 11. 贡献系数与防躺平

### 11.1 设计目标

解决两个矛盾：

- 不想让高层级用户躺平后长期白拿。
- 也不想让早期认真付出的人白白损失。

因此采用：

```text
贡献系数 + 缓冲期 + 快速恢复 + 历史贡献保护
```

### 11.2 贡献系数公式

```text
贡献系数 = 1.0 + 奖励加成 - 惩罚扣减
```

默认范围：

```text
最小值 = 0.3
最大值 = 1.5
```

最终计算：

```text
理想返利 = 基础层级返利 * 贡献系数
```

然后对所有受益人的理想返利做归一化，确保总额不超过返利池。

### 11.3 奖励加成

默认按最近 30 天统计。

自充奖励：

| 最近周期自充金额 | 加成 |
|---|---:|
| >= 100 | +0.1 |
| >= 200 | +0.2 |
| >= 500 | +0.3 |

拉新奖励：

| 最近周期新增有效下级 | 加成 |
|---|---:|
| >= 1 人 | +0.1 |
| >= 2 人 | +0.2 |
| >= 3 人 | +0.3 |

### 11.4 惩罚扣减

| 连续无贡献时间 | 建议系数 |
|---|---:|
| 30 天 | 0.7 |
| 60 天 | 0.5 |
| 90 天 | 0.3 |

无贡献定义：

- 最近周期没有自充。
- 最近周期没有新增有效下级。
- 没有其他被系统认定的有效推广行为。

### 11.5 历史贡献保护

必须做成开关。

开启后，历史贡献达到门槛的用户，不会被惩罚到过低。

示例：

```text
累计有效邀请 >= 10 人
或累计下级充值贡献 >= 3000 元
则最低贡献系数不低于 0.7
```

### 11.6 MVP 建议

防躺平机制建议第二阶段启用。

MVP 先做：

- 记录用户自充。
- 记录新增有效下级。
- 记录连续无贡献天数。
- 展示贡献系数配置，但默认关闭。

这样不会阻塞第一版上线，同时为后续开启做数据准备。

## 12. 归一化算法

### 12.1 为什么必须有

贡献系数可能让某些用户理想返利变高。

如果不归一化，可能超过返利池。

用户已明确选择：

> 使用归一化，保证每笔返利总额不超过平台设定返利池。

### 12.2 固定比例 + 贡献系数归一化

计算步骤：

1. 获取本次返利池。
2. 获取充值用户上级链路。
3. 根据层级配置取基础比例。
4. 如果实际层级少于配置层级，先对存在的层级做权重归一。
5. 得到每个受益人的基础返利。
6. 乘以贡献系数，得到理想返利。
7. 对所有理想返利再次归一化，使总额等于返利池。
8. 写入流水和余额。

公式：

```text
基础金额_i = 返利池 * 层级权重_i / 有效层级权重总和

理想金额_i = 基础金额_i * 贡献系数_i

最终金额_i = 理想金额_i / sum(理想金额) * 返利池
```

### 12.3 示例

返利池 15 元。

实际上级 3 人：

```text
第1级权重 60
第2级权重 20
第3级权重 10
有效权重合计 90
```

基础金额：

```text
第1级 = 15 * 60 / 90 = 10.00
第2级 = 15 * 20 / 90 = 3.33
第3级 = 15 * 10 / 90 = 1.67
```

贡献系数：

```text
第1级 = 1.2
第2级 = 1.0
第3级 = 0.7
```

理想金额：

```text
第1级 = 12.00
第2级 = 3.33
第3级 = 1.17
理想总额 = 16.50
```

最终归一化：

```text
第1级 = 12.00 / 16.50 * 15 = 10.91
第2级 = 3.33 / 16.50 * 15 = 3.03
第3级 = 1.17 / 16.50 * 15 = 1.06
合计 = 15.00
```

## 13. 核心业务流程

### 13.1 新用户绑定邀请关系

```text
用户访问推广链接
    |
    v
注册 / 登录
    |
    v
系统读取邀请码
    |
    v
校验邀请码是否有效
    |
    v
写入 referral_paths
    |
    v
生成该用户自己的邀请码
```

验收标准：

- 邀请关系不会丢失。
- 下级继续邀请时，层级自动延伸。
- 不能循环绑定。
- 不能自己邀请自己。

### 13.2 用户充值并触发返利

```text
用户充值成功
    |
    v
记录支付 / 额度变化
    |
    v
查询该用户直接上级和上级链路
    |
    v
判断里程碑奖励是否已用完
    |
    |-- 未用完 -> 计算并发放里程碑奖励
    |
    |-- 已用完 -> 执行正常多级分发
    |
    v
写 rebate_records
    |
    v
更新 rebate_balance
```

验收标准：

- 同一充值事件不会重复发放。
- 里程碑奖励次数准确。
- 多级分发金额总和等于返利池。
- 所有金额有流水可查。

### 13.3 用户提现

```text
用户提交提现申请
    |
    v
校验可提现余额
    |
    v
校验最低提现金额 / 次数 / 风控
    |
    v
冻结提现金额
    |
    v
管理员审核
    |
    |-- 通过 -> 打款并扣除冻结金额
    |
    |-- 拒绝 -> 解冻金额
    |
    v
记录提现流水
```

验收标准：

- 余额不会被重复提现。
- 审核拒绝后金额退回。
- 提现记录可导出。
- 管理员操作可追溯。

## 15. 配置项与 Tips 文案

后台配置项必须让运营者看得懂。

### 15.1 里程碑配置

| 配置项 | 默认值 | Tips |
|---|---:|---|
| 里程碑金额 | 100 | 新人累计充值每达到该金额，触发一次直接上级奖励 |
| 每次奖励金额 | 15 | 每次里程碑触发时，直接上级获得的返利金额 |
| 奖励次数 | 3 | 同一个新人最多触发多少次里程碑奖励 |
| 只奖励直接上级 | 开启 | 开启后，里程碑阶段第 2 级以上不获得奖励 |

### 15.2 多级分发配置

| 配置项 | 默认值 | Tips |
|---|---:|---|
| 分发模式 | 固定比例 | 固定比例更容易理解，衰减系数适合高级玩法 |
| 第 1 级比例 | 60 | 直接邀请人比例，应设置为最高 |
| 第 2 级比例 | 20 | 直接邀请人的上级比例 |
| 第 3 级比例 | 10 | 第 3 层上级比例 |
| 第 4 级比例 | 6 | 深层级建议较低 |
| 第 5 级比例 | 4 | 深层级建议较低 |
| 最大计算层级 | 5 | 超过该层级不参与返利 |
| 有效层级归一化 | 开启 | 实际上级少于配置层级时，把返利池重新分配给已有上级 |

### 15.3 防躺平配置

| 配置项 | 默认值 | Tips |
|---|---:|---|
| 启用贡献系数 | 关闭 | 开启后，用户活跃度会影响最终返利 |
| 奖罚周期 | 30 天 | 系统统计最近多少天的自充和拉新 |
| 最低贡献系数 | 0.3 | 长期不活跃用户最低能拿到的比例 |
| 最高贡献系数 | 1.5 | 高活跃用户最高奖励倍率 |
| 历史贡献保护 | 关闭 | 保护曾经贡献较大的推广员，避免惩罚过重 |
| 快速恢复 | 开启 | 用户重新活跃后，贡献系数可恢复到正常水平 |

### 15.4 提现配置

| 配置项 | 默认值 | Tips |
|---|---:|---|
| 最低提现金额 | 待定 | 金额过低会增加审核和手续费成本 |
| 提现审核 | 人工审核 | MVP 建议人工审核，降低资金风险 |
| 提现冻结期 | 待定 | 新获得返利可冻结一段时间后提现 |
| 每日提现次数 | 待定 | 防止频繁提现和异常操作 |
| 支付宝账号校验 | 开启 | 用户提现前需要填写并确认支付宝账号 |

## 18. 验收标准

### 18.1 邀请关系

- 用户通过推广链接注册后，能正确绑定直接上级。
- 下级继续邀请别人时，系统能正确形成多级链路。
- 管理后台能查看某个用户的上级链路和下级树。

### 18.2 里程碑奖励

- 新人累计充值达到 100 元时，直接上级获得奖励。
- 累计充值跨多个里程碑时，不漏发、不重复发。
- 达到最大奖励次数后，不再触发里程碑奖励。
- 里程碑阶段第 2 级以上不获得奖励。

### 18.3 多级分发

- 里程碑结束后，后续充值进入多级分发。
- 第 1 级获得最多。
- 实际层级不足时，返利池能按已有上级归一化分完。
- 每次分发总金额等于返利池。

### 18.4 余额与提现

- 返利进入独立返利余额。
- 用户申请提现后，可提现余额减少或冻结。
- 审核拒绝后金额退回。
- 审核通过后记录提现流水。

### 18.5 审计与风控

- 每笔返利有来源事件。
- 每笔提现有审核记录。
- 管理员修改配置有操作记录。
- 同一事件不会重复发放返利。

## 19. 仍需最终确认的问题

以下问题建议下一轮继续研讨并定稿：

1. 里程碑奖励次数默认是 3 次还是 5 次。
2. 里程碑奖励是否允许不同次数不同金额，例如第 1 次 30，第 2 次 15。
3. MVP 是否采用“有效层级归一化分完返利池”作为最终口径。
4. 多级分发默认比例是否确定为 `60 / 20 / 10 / 6 / 4`。
5. 最大计算层级默认 5 级是否足够。
6. 贡献系数是否第一版只记录不启用。
7. 人工提现第一版最低提现金额设置多少。
8. 管理员手动给 Sub2API 用户加额度是否也要触发返利。
9. 活动返利是否第一版就支持。
10. 支付宝充值是否第一版接入，还是第二阶段接入。

## 20. 推荐定稿版本

如果需要尽快进入实施，建议采用以下版本作为第一版定稿：

```text
系统形态：
Sub2Rebate 独立系统，不修改 Sub2API 核心代码。

返利池：
默认每充值 100 元，返利池 15 元。

里程碑：
里程碑金额 100 元。
每次奖励 15 元。
奖励次数 3 次。
只奖励直接上级。

多级分发：
里程碑结束后启用。
固定比例：60 / 20 / 10 / 6 / 4。
最大 5 级。
实际层级不足时，对已有上级归一化分完返利池。

贡献系数：
第一版只记录数据，后台预留配置，默认不开启。

提现：
第一版人工审核，支付宝手动打款。
返利余额独立于 Sub2API 额度。

风控：
防重复发放、黑名单、最低提现金额、完整流水。
```

该版本好处：

- 能快速上线验证推广模型。
- 算法足够清晰，用户容易理解。
- 不会被防躺平、自动支付等复杂模块拖慢第一版。
- 后续可以平滑升级到支付宝自动化和贡献系数。

---

## 21. 最终一句话

Sub2Rebate 第一版应先完成“邀请树 + 新人里程碑奖励 + 后续多级返利 + 独立余额 + 人工提现 + 完整流水”的闭环；等业务跑通后，再逐步加入支付宝自动充值提现、贡献系数、防躺平、活动配置和更复杂的衰减算法。
