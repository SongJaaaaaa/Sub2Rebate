# Invite 模块：邀请关系与邀请树

## 1. 模块目标

负责维护用户之间的邀请关系，生成树状裂变结构，并为里程碑奖励和多级返利提供上级链路。

## 2. 模块边界

本模块负责：

- 邀请码。
- 推广链接。
- 直接上级绑定。
- 邀请路径。
- 邀请树查询。
- 防止循环邀请。

本模块不负责：

- 返利金额计算。
- 提现。
- 支付。

## 3. 核心概念

| 概念 | 说明 |
|---|---|
| inviter | 邀请人，直接上级 |
| invitee | 被邀请人，下级 |
| referral path | 从根用户到当前用户的路径 |
| depth | 用户在邀请树中的深度 |
| structure chain | 历史结构链路，不因中间节点失效而自动重排 |
| rebate eligibility | 返利资格，决定节点是否能获得新返利 |

## 4. 建议数据结构

| 表 | 说明 |
|---|---|
| referral_paths | 邀请路径表 |

核心字段：

```text
user_id
parent_id
path
depth
root_inviter_id
invite_code
created_at
updated_at
```

## 5. 核心流程

```text
用户访问推广链接
-> 注册 / 登录
-> 校验邀请码
-> 绑定直接上级
-> 生成 path 和 depth
-> 生成自己的邀请码
```

## 5.1 失效节点规则

邀请树表示历史关系，不因为中间节点失效而自动重排。

示例：

```text
A
└── B
    └── C
        └── D
```

当 B 被 Sub2API 删除、失联，或后续被防躺平机制判定为返利失效时：

- C 不上浮到 A 名下。
- C 及其以下节点仍保留在 B 的下级。
- `parent_user_id` 和 `path` 不自动改写。
- 管理端邀请树中 B 置灰展示。
- B 可以继续登录，除非账号状态被明确封禁。
- B 不再获得新的返利，返利计算由 Rebate 模块按配置处理。

实现时需要区分两类链路：

```text
ancestorIds()
    返回完整结构上级链路，用于树、审计、复盘。

rebateAncestorIds() / eligibleAncestorIds()
    返回返利计算需要的链路，并带上失效节点状态或按配置过滤。
```

不要通过重写 `path` 的方式处理失效节点。
