# 决策记录

> 记录重要技术和产品选择。每条决策都要写清背景、选择、原因和影响。

## ADR-001：采用独立 Sub2Rebate 系统，而不是直接修改 Sub2API

日期：2026-06-13

背景：

Sub2API 原生邀请返利机制无法覆盖管理员手动加额度等场景，并且后续需求包括多级分销、支付宝提现、风控、活动配置等复杂业务。

决策：

建设独立 Sub2Rebate 系统，不直接修改 Sub2API 核心代码作为第一方案。

原因：

- 避免 Sub2API 升级覆盖自定义代码。
- 返利、提现、风控属于独立业务域。
- 后续扩展更灵活。

影响：

- 需要维护独立数据库表。
- 需要设计与 Sub2API 的集成方式。
- 系统边界更清晰。

## ADR-002：前端采用 Vue 3 + Vite

日期：2026-06-13

背景：

用户熟悉 Vue，希望后续维护方便，并且需要独立用户中心和推广中心。

决策：

用户前端采用 Vue 3 + Vite。

原因：

- 用户熟悉，维护成本低。
- 用户中心不强依赖 SSR。
- Vite 开发体验好。
- 与 Laravel API 解耦清晰。

影响：

- SEO 型推广落地页后续如果有强需求，可单独补 Nuxt 或静态页。

## ADR-003：管理后台采用 Filament

日期：2026-06-13

背景：

系统需要大量表格、表单、配置、审核和流水页面。

决策：

管理后台采用 Filament。

原因：

- 与 Laravel 集成紧密。
- 快速生成资源管理、表单、表格、筛选、审核页面。
- 适合财务和运营后台。

影响：

- 管理后台不用单独开发 Vue。
- 用户前端和管理后台分离。

## ADR-005：Sub2API 手动加额度也需触发返利

日期：2026-06-13

背景：

Sub2Rebate 有自己的充值入口，但 Sub2API 管理员也可能手动给用户加额度。

决策：

两个来源的额度变化都需要触发返利引擎：

- Sub2Rebate 自身充值/加额度：直接触发。
- Sub2API 管理员手动加额度：通过监控 Sub2API 数据变化触发。

原因：

- 用户充值可能走旧入口或新入口。
- 不监控 Sub2API 的额度变化会导致漏发返利。

影响：

- Integration 模块需要设计对 Sub2API 额度变化的监控方案。
- 需要确认 Sub2API 数据库结构后才能实现。

## ADR-004：第一版暂不启用贡献系数

日期：2026-06-13

背景：

防躺平机制重要，但会增加规则复杂度和解释成本。

决策：

第一版记录贡献相关数据，但默认不启用贡献系数自动影响返利。

原因：

- 降低第一版复杂度。
- 先跑通返利闭环。
- 后续有数据后再开启更稳。

影响：

- 需要预留贡献数据字段。
- 第二阶段开启时不需要重构核心流水。

## ADR-006：用户认证复用 Sub2API 账号，区分管理员和普通用户

日期：2026-06-13

背景：

Sub2Rebate 需要用户登录才能使用推广、返利、提现功能，同时管理员需要独立的后台权限。

决策：

- 复用 Sub2API 用户表作为用户数据源。
- Sub2Rebate 读取 Sub2API 用户表验证身份。
- Sub2Rebate 签发自己的 token（Laravel Sanctum）。
- 区分两种角色：管理员（admin）和普通用户（user）。
- 管理员通过 Filament 后台登录，普通用户通过 Vue 前端登录。

原因：

- 用户不需要重复注册，降低使用门槛。
- Sub2Rebate 控制自己的会话，不依赖 Sub2API 的 token。
- 角色分离保证普通用户无法访问管理功能。

影响：

- 需要确认 Sub2API 用户表的账号密码字段结构。
- Sub2Rebate 需要一张角色/权限表标记哪些用户是管理员。
- 用户在 Sub2API 注册后即可登录 Sub2Rebate，无需二次注册。

## ADR-007：多级分发采用衰减系数模式，无限级金字塔结构

日期：2026-06-13

背景：

多级分发需要确定分配算法和层级策略。固定比例模式（60/20/10/6/4）需要手动配置每一级，层级固定为 5 级。

决策：

- 采用衰减系数模式。
- 默认衰减系数 = 0.4。
- 不设最大计算层级，按实际邀请链路的所有上级计算（无限级金字塔）。
- 所有有效上级都参与分配，通过衰减自然递减。
- 返利池全部分配给有效上级，不留归平台。

原因：

- 层级可动态扩展，不需要手动配置每一级比例。
- 直接上级永远拿最多，越往上自然递减。
- 适应实际推广场景中层级深浅不一的情况。
- 只需一个参数即可控制分配曲线。

影响：

- 后台配置更简单（只配衰减系数）。
- 深层级用户分到的金额很小但不为零。
- 需要考虑精度和尾差处理。

## ADR-008：里程碑奖励次数默认 2 次

日期：2026-06-13

背景：

里程碑奖励次数影响直接上级的初期激励力度和进入多级分发的速度。

决策：

默认奖励次数 = 2 次。

原因：

- 2 次里程碑奖励足以激励拉新。
- 更快进入多级分发，上级链路中更多人能更早受益。
- 可通过后台配置调整。

影响：

- 新人累计充值 200 元后即进入多级分发。
- 直接上级在里程碑阶段获得 30 元（2 × 15）。
