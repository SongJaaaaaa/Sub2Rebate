# 决策记录

> 记录重要技术和产品选择。每条决策都要写清背景、选择、原因和影响。

## ADR-001：采用独立 Sub2Rebate 系统，而不是直接修改 Sub2API

日期：2026-06-13

背景：

Sub2API 原生邀请返利机制无法覆盖管理员手动加额度等场景，并且后续需求包括多级分销、支付宝提现、风控、活动配置等复杂业务。

决策：

建设独立 Sub2Rebate 系统，不直接修改 Sub2API 核心代码作为第一方案。

原因：

- 避免 Sub2API 升级覆盖自定义代码。
- 返利、提现、风控属于独立业务域。
- 后续扩展更灵活。

影响：

- 需要维护独立数据库表。
- 需要设计与 Sub2API 的集成方式。
- 系统边界更清晰。

## ADR-002：前端采用 Vue 3 + Vite

日期：2026-06-13

背景：

用户熟悉 Vue，希望后续维护方便，并且需要独立用户中心和推广中心。

决策：

用户前端采用 Vue 3 + Vite。

原因：

- 用户熟悉，维护成本低。
- 用户中心不强依赖 SSR。
- Vite 开发体验好。
- 与 Laravel API 解耦清晰。

影响：

- SEO 型推广落地页后续如果有强需求，可单独补 Nuxt 或静态页。

## ADR-003：管理后台采用 Filament

日期：2026-06-13

背景：

系统需要大量表格、表单、配置、审核和流水页面。

决策：

管理后台采用 Filament。

原因：

- 与 Laravel 集成紧密。
- 快速生成资源管理、表单、表格、筛选、审核页面。
- 适合财务和运营后台。

影响：

- 管理后台不用单独开发 Vue。
- 用户前端和管理后台分离。

## ADR-005：Sub2API 手动加额度需进入明确事件入口后才触发返利

日期：2026-06-13

背景：

Sub2Rebate 有自己的充值入口，但 Sub2API 管理员也可能手动给用户加额度。由于 Sub2API 余额会被充值、消费扣费、退款和管理员调整共同影响，不能把余额变化本身当成可靠充值事件。

决策：

两个来源的充值/加额度都应进入 Sub2Rebate 的明确事件入口后再触发返利引擎：

- Sub2Rebate 自身充值/加额度：直接触发。
- Sub2API 兑换码/支付订单：通过扫描 `redeem_codes`、`payment_orders`、`payment_audit_logs` 生成幂等事件。
- Sub2API 管理员手动加额度：原则上统一改走兑换码/订单/Sub2Rebate 后台补录入口；如果仍直接修改 `users.balance`，只做对账提示和人工确认，不自动发放返利。

原因：

- 用户充值可能走旧入口或新入口。
- 返利涉及金额，必须有来源、幂等键、操作者、备注和审计。
- 单纯监控余额变化会把 API 消费扣费、退款和手动调额混在一起。

影响：

- Integration 模块需要设计对 Sub2API 外部充值/兑换事件的扫描方案，并保留余额快照用于对账。
- 源码层面已确认 `payment_orders`、`redeem_codes`、`payment_audit_logs` 可作为更可靠的充值/兑换事件源。
- 生产库表结构已通过 SSH 只读复核，`sub2rebate_ro` 只读账号已创建并验证。

## ADR-004：第一版暂不启用贡献系数

日期：2026-06-13

背景：

防躺平机制重要，但会增加规则复杂度和解释成本。

决策：

第一版记录贡献相关数据，但默认不启用贡献系数自动影响返利。

原因：

- 降低第一版复杂度。
- 先跑通返利闭环。
- 后续有数据后再开启更稳。

影响：

- 需要预留贡献数据字段。
- 第二阶段开启时不需要重构核心流水。

## ADR-006：用户认证复用 Sub2API 账号，区分管理员和普通用户

日期：2026-06-13

背景：

Sub2Rebate 需要用户登录才能使用推广、返利、提现功能，同时管理员需要独立的后台权限。

决策：

- 复用 Sub2API 用户表作为用户数据源。
- Sub2Rebate 读取 Sub2API 用户表验证身份。
- Sub2Rebate 签发自己的 token（Laravel Sanctum）。
- 区分两种角色：管理员（admin）和普通用户（user）。
- 管理员通过 Filament 后台登录，普通用户通过 Vue 前端登录。

原因：

- 用户不需要重复注册，降低使用门槛。
- Sub2Rebate 控制自己的会话，不依赖 Sub2API 的 token。
- 角色分离保证普通用户无法访问管理功能。

影响：

- 源码层面已确认 Sub2API 用户表包含 `email`、`password_hash`、`role`、`balance`、`status` 等字段。
- Sub2Rebate 需要一张角色/权限表标记哪些用户是管理员。
- 用户在 Sub2API 注册后即可登录 Sub2Rebate，无需二次注册。

## ADR-007：多级分发采用衰减系数模式，无限级金字塔结构

日期：2026-06-13

背景：

多级分发需要确定分配算法和层级策略。固定比例模式（60/20/10/6/4）需要手动配置每一级，层级固定为 5 级。

决策：

- 采用衰减系数模式。
- 默认衰减系数 = 0.4。
- 不设最大计算层级，按实际邀请链路的所有上级计算（无限级金字塔）。
- 所有有效上级都参与分配，通过衰减自然递减。
- 返利池全部分配给有效上级，不留归平台。

原因：

- 层级可动态扩展，不需要手动配置每一级比例。
- 直接上级永远拿最多，越往上自然递减。
- 适应实际推广场景中层级深浅不一的情况。
- 只需一个参数即可控制分配曲线。

影响：

- 后台配置更简单（只配衰减系数）。
- 深层级用户分到的金额很小但不为零。
- 需要考虑精度和尾差处理。

## ADR-008：里程碑奖励次数默认 2 次

日期：2026-06-13

背景：

里程碑奖励次数影响直接上级的初期激励力度和进入多级分发的速度。

决策：

默认奖励次数 = 2 次。

原因：

- 2 次里程碑奖励足以激励拉新。
- 更快进入多级分发，上级链路中更多人能更早受益。
- 可通过后台配置调整。

影响：

- 新人累计充值 200 元后即进入多级分发。
- 直接上级在里程碑阶段获得 30 元（2 × 15）。

## ADR-009：技术栈版本确认为 Laravel 12 + Filament 3

日期：2026-06-13

背景：

原规划写 Laravel 13 + Filament 5，但截至 2026 年 6 月，Laravel 12 和 Filament 3 是最新稳定版本。

决策：

- 后端使用 Laravel 12。
- 管理后台使用 Filament 3。

原因：

- 稳定版本，生态成熟，插件和社区支持完善。
- 避免使用未发布或不稳定版本导致开发受阻。

影响：

- 全局文档中的版本号统一为 Laravel 12 / Filament 3。

## ADR-010：Sub2API 集成采用共享数据库方式

日期：2026-06-13

背景：

需要从 Sub2API 感知兑换码、支付订单和支付审计等明确充值来源。Sub2API 无 webhook 推送，需要选择监控方案；管理员手动加额度不能仅靠余额变化自动触发返利。

决策：

- Sub2Rebate 直接读取 Sub2API 数据库中的充值、兑换和支付审计记录。
- 第一优先级扫描 `payment_orders`、`redeem_codes`、`payment_audit_logs`，触发返利引擎。
- `users.balance` / `users.total_recharged` 只作为对账快照和人工确认线索，不作为第一版稳定事件源。
- 上游账号列表和上游账号用量通过 Sub2API Admin API 调阅，优先使用 `x-api-key`；它不属于用户账号体系。

原因：

- 共享数据库最简单直接，无需 Sub2API 改代码。
- 第一版追求快速上线，后续可升级为事件推送。
- 余额字段会被充值、消费扣款、退款、管理员调额同时影响，单独监控差值容易误发或漏发返利。

影响：

- Sub2Rebate 需要配置 Sub2API 的数据库连接（只读）。
- 生产库表结构已完成只读复核。
- 定时任务需要记录上次扫描位置，避免重复触发。

## ADR-011：提现最低金额 50 元

日期：2026-06-13

背景：

需要确定提现门槛，平衡用户体验和运营成本。

决策：

- 提现最低金额默认 50 元，可通过后台配置修改。

原因：

- 避免频繁小额提现增加人工审核负担。
- 50 元门槛对推广者来说不难达到。

影响：

- Config 模块新增 `withdraw_min_amount` 配置项，默认 50。
- 前端提现页面需要提示最低金额。

## ADR-012：充值返利事件必须来自明确充值来源，余额快照只做对账

日期：2026-06-13

背景：

Sub2API 的 `users.balance` 会随充值增加，也会随 API 消费、退款、管理员调整而减少或变化。单纯读取余额差值会把消费扣费、退款、手动调额和真实充值混在一起，容易造成返利统计混乱。

决策：

- 返利触发必须进入 Sub2Rebate 的 `rebate_events` 事件入口。
- 第一优先级读取 Sub2API 的明确充值/兑换来源：`redeem_codes`、`payment_orders`、`payment_audit_logs`。
- Sub2API 管理员手动加额度如果要自动触发返利，原则上应统一走可追踪的兑换码/订单/新系统管理入口。
- `users.balance` / `users.total_recharged` 快照只用于对账、异常提示或人工确认，不能直接自动发放返利。
- 长期建议所有涉及充值、加额度、返利触发的操作都尽量在 Sub2Rebate 管理后台完成，由后台写入审计和幂等事件，再同步或调用 Sub2API。

原因：

- 余额会持续减少，无法代表充值事件本身。
- 金额和返利涉及财务，必须有幂等键、来源、操作者、备注和审计。
- 将手动加额度收敛到新系统，可以避免绕过权限、审计和返利规则。

影响：

- B5 Payment 事件入口必须先实现幂等、安全校验和审计字段。
- B11 Sub2API 扫描器不能把余额差值作为自动返利依据。
- 后台手动创建充值事件、调整返利余额、提现审核等敏感操作必须要求管理员权限和备注。

## ADR-013：金额口径和兑换比例后台可配置

日期：2026-06-13

背景：

当前 Sub2API 业务口径是 1 人民币 = 1 刀/额度，但长期可能调整币种、额度和人民币之间的换算关系。

决策：

- 充值金额、返利池、里程碑计算使用 Sub2Rebate 的标准金额口径。
- Sub2API 额度和人民币/返利金额之间的换算比例必须进入配置中心。
- 默认配置可按当前口径设置为 `payment.cny_to_credit_rate = 1`。

原因：

- 避免把当前 1:1 口径写死到业务代码。
- 后续调整价格、额度或币种时只改配置，不改返利引擎。

影响：

- B3 配置中心新增支付/金额换算配置。
- B5 创建充值事件时必须保存原始来源金额、标准金额、币种/单位和换算配置快照。

## ADR-014：管理权限优先跟随 Sub2API 角色，并允许本系统补充限制

日期：2026-06-13

背景：

Sub2Rebate 账号池设想与 Sub2API 用户账号池共用，Sub2API 已有 `admin` / `user` 等角色信息。

决策：

- Filament 后台登录优先依据 Sub2API `users.role = admin`。
- 本系统保留 `sub2_user_roles` 作为补充权限表，用于更细粒度开关、冻结后台权限或后续扩展。
- 涉及金额、提现、配置、事件补录的后台操作必须校验管理员权限。

原因：

- 与 Sub2API 账号池保持一致，减少重复维护。
- 本系统涉及财务，需要保留自己的二次权限控制能力。

影响：

- B10 后台登录策略需要读取 Sub2API 角色和本地补充权限。
- 普通用户即使能登录用户端，也不能访问后台和任何管理 API。
