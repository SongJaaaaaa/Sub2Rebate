- 不直接修改 Sub2API 源码。

## 2. 原始业务痛点

用户当前已经部署 Sub2API，并发现：

- 邀请关系能建立。
- 被邀请用户会出现在邀请列表。
- 但是返利只在“兑换码兑换额度”时触发。
- 管理员后台手动给用户设置额度时，不触发返利。
- 这导致推广员看不到应得返利。

用户关注的不是单纯代码 bug，而是推广体系中的关键痛点：

- 下级推广动力不足。
- 管理员手动发额度时无法纳入返利。
- 原生返利机制无法满足运营需求。
- 如果要大规模推广，必须有一套更完整的返利系统。

---

## 3. Sub2API 原生机制理解

对话中先确认了 Sub2API 的基本定位：

- Sub2API 是 AI API 中转网关。
- 可将 Claude Pro、ChatGPT Plus / Team、Gemini Advanced 等网页订阅账号池化。
- 对外转换成 OpenAI 兼容 API。
- 可用于 Cursor、Windsurf、Claude Code、OpenCode 等工具。
- 支持账号共享、拼车、负载均衡、额度统计等。

对原生返利机制的判断：

```text
兑换码
-> Recharge 事件
-> Referral / Rebate 逻辑
-> 邀请人获得返利
```

但管理员手动加额度可能走的是另一条路径：

```text
Admin Update Balance
-> 直接修改用户额度
-> 不经过 Recharge
-> 不触发 Referral / Rebate
```

因此核心问题是：

> 所有额度增加没有统一进入返利计算入口。

## 4. 第一轮方案：直接补 Sub2API 返利逻辑

最初建议是从 Sub2API 内部修复：

- 找到 `AddBalance`、`UpdateBalance`、`SetQuota`、`Recharge` 等入口。
- 把所有加额度行为统一到一个服务方法。
- 在统一方法中检查是否有邀请人。
- 计算返利并写入返利余额。

伪代码思路：

```go
func (s *BalanceService) AddBalance(ctx context.Context, userID uint, amount float64, source string) error {
    err := s.repo.AddBalance(userID, amount)
    if err != nil {
        return err
    }

    if referrerID, exists := s.getReferrer(userID); exists {
        rebateAmount := amount * s.getRebateRate()
        s.referralService.AwardRebate(ctx, referrerID, userID, rebateAmount, source)
    }

    return nil
}
```

这种方案的优点：

- 改动目标直接。
- 能让管理员加额度也触发返利。
- 逻辑在 Sub2API 内部闭环。

但用户马上提出关键问题：

> 这样写的话，每次 Sub2API 更新，是不是代码就没了？

这个问题非常关键，直接推动方案转向。

## 5. 自改源码的维护问题

讨论中确认：

直接改 Sub2API 源码会有明显风险：

- 上游更新会覆盖修改。
- `git pull` 时可能冲突。
- 重新拉 Docker 镜像会丢失自定义逻辑。
- 每次升级都要重新打补丁。
- 返利逻辑越复杂，冲突越难维护。

曾讨论过几个替代方式：

### 5.1 Fork 自己的仓库

做法：

- Fork 官方 Sub2API。
- 创建自己的功能分支。
- 在分支中修改返利逻辑。
- 部署自己的镜像。
- 上游更新时 merge / rebase。

优点：

- 修改不会直接丢。
- 比服务器上直接改源码更规范。

缺点：

- 仍然要维护冲突。
- 后续业务越来越复杂后，Sub2API 会被越改越重。

### 5.2 Patch 文件

做法：

```text
git diff > rebate-improvement.patch
git apply rebate-improvement.patch
```

优点：

- 轻量。
- 不必维护完整 Fork。

缺点：

- 冲突时仍要人工处理。
- 不适合长期复杂功能。

### 5.3 给官方提 PR

思路：

- 将“管理员加额度是否触发返利”做成可配置开关。
- 默认关闭。
- 提交给官方。

优点：

- 官方接收后维护成本最低。

缺点：

- 官方是否接收不可控。
- 用户后续需求已经远超一个开关。

## 6. 关键转向：独立外部返利系统

用户提出新的想法：

> 因为项目部署在自己的服务器中，可以写一套邀请返利系统。  
> 用 Sub2API 管理员账号登录，获取所有 Sub2 账号。  
> 对用户进行添加或修改额度。  
> 再调用接口去改用户返利信息。

这个想法本质是：

> 用一个独立系统包裹 Sub2API 的管理员操作，实现外部监控和返利补偿。

讨论后形成系统名称：

```text
Sub2Rebate
```

Sub2Rebate 的定位：

- 独立于 Sub2API。
- 不改 Sub2API 核心代码。
- 自己维护邀请关系、返利规则、返利余额、提现和流水。
- 通过数据库或接口感知 Sub2API 用户充值/额度变化。

## 7. DB 监听 vs Admin API 轮询

外部系统与 Sub2API 集成时，讨论了两个方案。

### 7.1 方案 A：直接连接 PostgreSQL

做法：

- Sub2Rebate 和 Sub2API 使用同一个 PostgreSQL。
- 监听 users、user_balances 或相关表。
- 使用 PostgreSQL Trigger + NOTIFY，或定时轮询。
- 检测用户额度增加。
- 查询邀请关系。
- 计算返利。
- 写入 Sub2Rebate 自己的返利记录和余额。

优点：

- 稳定可靠。
- 几乎实时。
- 不依赖管理员 token。
- 不怕 API 限流。
- 不怕登录过期。
- 表结构通常比接口更稳定。

缺点：

- 需要了解 Sub2API 表结构。
- 大版本升级时字段变化要适配。
- 如果直接写数据库，需要注意事务和安全。

### 7.2 方案 B：Admin API 轮询

做法：

- 用管理员账号登录。
- 获取 token。
- 轮询 `/api/admin/users` 或类似接口。
- 对比用户额度快照。
- 发现额度增加后触发返利。

优点：

- 更像走官方接口。
- 不直接读写数据库。

缺点：

- token 可能过期。
- API 可能变动。
- 轮询有延迟。
- 用户多时压力大。
- 可能被限流。

### 7.3 最终倾向

多轮比较后，推荐方案 A：

> 直接连接 PostgreSQL，使用 Trigger + NOTIFY 或轻量轮询。

理由：

- 最稳定。
- 最实时。
- 后期维护更容易。
- 更适合用户自建服务器场景。

## 11. 支付宝充值与提现条件

对话中讨论了支付宝接入需要的条件。

### 11.1 支付宝扫码充值

需要：

- 企业支付宝账号，个体户也可能可行。
- 支付宝开放平台应用。
- 开通当面付、电脑网站支付、手机网站支付等能力。
- 域名。
- HTTPS。
- 支付回调地址。
- 可能需要备案。

充值流程：

```text
用户在新系统发起充值
-> 生成支付宝二维码或支付链接
-> 用户扫码付款
-> 支付宝回调
-> 系统确认支付成功
-> 增加用户额度
-> 触发返利逻辑
```

### 11.2 支付宝提现

提现流程：

```text
用户申请提现
-> 系统校验可提现余额
-> 管理员审核
-> 调用支付宝转账接口或人工打款
-> 扣减返利余额
-> 记录提现流水
```

需要考虑：

- 最低提现金额。
- 手续费。
- 提现次数限制。
- 冻结期。
- 黑名单。
- 审核流程。
- 支付宝账号校验。

## 12. 多级分销层级识别

用户提出关键问题：

> 如果每个人都有自己的邀请码，下级直接拿邀请码继续邀请，怎么知道他是第几级？

解决方案：

> Sub2Rebate 自己维护一套完整邀请链路，不依赖 Sub2API 原生平级邀请码逻辑。

推荐使用 Materialized Path：

```text
user_id          当前用户
parent_id        直接上级
path             路径，例如 "1001.1005.1012"
depth            层级深度
root_inviter_id  顶级邀请人
```

绑定流程：

```text
新人注册时提交邀请码
-> 查询邀请码对应用户作为 parent_id
-> 查询 parent 的 path 和 depth
-> 新用户 path = parent.path + "." + new_user_id
-> 新用户 depth = parent.depth + 1
-> 写入 referral_paths
```

这个方案支持：

- 无限层级。
- 多叉树。
- 快速查询上下级。
- 动态计算真实邀请链路。

树状示例：

```text
A
├── B
│   ├── D
│   └── E
└── C
    ├── F
    └── G
```

当 D 充值，只计算：

```text
D -> B -> A
```

当 F 充值，只计算：

```text
F -> C -> A
```

## 13. 返利池最初讨论

用户明确了一个重要限制：

> 我的利润中，100 块钱我最大让利 15，也就是拿 15 元进行分发。

最初讨论过：

```text
总返利池 = 充值金额 * 15%
```

也就是：

```text
充值 100 元 -> 返利池 15 元
```

曾经讨论过“剩余归平台”的方式：

```text
第1级 55% = 8.25
第2级 25% = 3.75
第3级 12% = 1.80
第4级 6%  = 0.90
第5级 2%  = 0.30
```

但用户后面明确纠正：

> 我可以拿出 15 块完全去分配，不需要考虑剩多少归平台。

因此后续方案应遵守：

- 返利池默认 100 返 15。
- 返利池可以调整。
- 返利池应尽量全部分配。
- 不要再默认设计“剩余归平台”作为主逻辑。

## 14. 层级分发最初方案

最初多级分发比例曾讨论：

方案一：

```text
第1级：60% = 9.00
第2级：20% = 3.00
第3级：10% = 1.50
第4级：6%  = 0.90
第5级：4%  = 0.60
总计：15.00
```

方案二：

```text
第1级：70% = 10.50
第2级：15% = 2.25
第3级：8%  = 1.20
第4级：4%  = 0.60
第5级：3%  = 0.45
总计：15.00
```

用户认可的核心方向：

> 努力的人赚得多，上级少一点。  
> 直接拉人的人拿最多，上面的少。  
> 这样可以一直递增，形成正向激励。

## 15. 第四级及以上如何处理

用户问：

> 如果第四级呢？现在情况第三级邀请一个充值 100，利润怎么分配？

当时用例：

```text
A -> B -> C -> D
D 充值 100
```

相对 D：

```text
C 是第1级
B 是第2级
A 是第3级
```

如果只有 3 级上级，则只给 C、B、A。

关键点：

- 系统根据实际邀请链路动态计算。
- 不强行固定 5 级。
- 没有第 4 级，就不计算第 4 级。
- 具体未分配部分如何处理，后续需要定规则。

用户后续更倾向：

> 返利池全部分配，不需要考虑剩余归平台。

因此合理规则应是：

- 如果配置了 5 级，但实际只有 3 级，可对实际存在的 3 级重新归一化。
- 让 15 元全部分配给已有上级。

## 16. 树状裂变是否支持

用户问：

> 你考虑树状结构了吗？会有分叉裂变的形式。

答复中确认：

- 必须支持树状裂变。
- 一个用户可以发展多个下级。
- 每个下级又可以继续发展自己的下级。
- 每个充值事件只沿充值人的上级路径计算。

树状结构是分销系统的核心，不是可选项。

## 17. 错误方案纠正：固定大额返利池

有一轮讨论中，曾误解为：

> 用户希望设置固定 100 或 200 的返利池。

给出过类似：

```text
利润池 = 100 元
第1级 65 元
第2级 20 元
...
```

用户立即纠正：

> 你这样完全算错了。  
> 我是别人充值 100，我可以拿出 15 块完全去分配。  
> 不需要考虑剩多少归平台。  
> 其中奖励首次拉人头机制，是拉的人头充值了 100，这就有 15 块奖励，只给这个人，上级不给。

这个纠正非常重要，最终规则必须以此为准。

## 18. 首次拉人头奖励：最终关键规则

用户明确提出：

> 奖励首次拉人头机制是：拉的人头充值了 100 块，然后有 15 块奖励，只给直接拉人的人，上级不给。

随后进一步澄清：

> 规则需要基于拉人后，新人累计充值 100 块才可以有 15 的奖励。  
> 受奖励次数比如设置 3 次，那么就是 15 * 3 的金额。  
> 也就是拉的人需要充值够 300。

因此最终应理解为：

```text
新人累计充值达到一个里程碑
-> 直接上级获得一次奖励
-> 奖励次数可配置
-> 次数用完后进入正常多级分发
```

默认示例：

```text
里程碑金额 = 100
每次奖励 = 15
奖励次数 = 3
```

则：

```text
新人累计充值 100 -> 直接上级 +15
新人累计充值 200 -> 直接上级 +15
新人累计充值 300 -> 直接上级 +15
之后再充值 -> 进入多级分发
```

注意：

- 是累计充值，不是单笔充值。
- 只奖励直接上级。
- 第 2 级以上不参与。
- 100 和 15 都必须可配置。
- 奖励次数必须可配置。

## 19. 里程碑规则细化

用户希望：

> 里程碑、15 这些都可以调整。  
> 默认 100 返 15，里程碑也可以设置。  
> 初始奖励里程碑完成后，就计算公式，每一层级的返利算法。

因此里程碑配置应包括：

| 配置项 | 默认值 | 说明 |
|---|---:|---|
| 里程碑金额 | 100 | 新人累计充值达到多少触发 |
| 每次奖励金额 | 15 | 每次触发给直接上级多少 |
| 奖励次数 | 3 | 同一个新人最多触发几次 |
| 是否只奖励直接上级 | 是 | 上级链路不参与 |

示例：

```text
里程碑金额 = 100
奖励金额 = 15
奖励次数 = 3

新人累计充值 60 -> 不触发
再充值 50，累计 110 -> 触发第 1 次
再充值 100，累计 210 -> 触发第 2 次
再充值 100，累计 310 -> 触发第 3 次
再充值 100，累计 410 -> 正常多级分发
```

待进一步确认：

- 如果一次充值跨多个里程碑，是否一次发多次奖励。
- 如果一次充值刚好用完里程碑次数，超出部分是否立刻进入多级分发。
- MVP 是否简化为“下一笔充值再进入多级分发”。

## 21. 衰减系数讨论

曾讨论用衰减系数实现层级递减。

示例：

```text
第1级：9.00
衰减系数：0.4
第2级：9.00 * 0.4 = 3.60
第3级：3.60 * 0.4 = 1.44
第4级：1.44 * 0.4 = 0.576
第5级：0.576 * 0.4 = 0.2304
```

优点：

- 自动递减。
- 层级多时不用手动配置每一级。
- 可支持动态链路。

问题：

- 总额可能不等于返利池。
- 需要归一化。
- 用户理解成本比固定比例高。

后续用户强调：

> 归一化 + 衰减系数就是核心算法，这里一定要处理好。

因此可保留两种方案：

1. 固定比例模式：简单直观，适合 MVP。
2. 衰减系数模式：更灵活，适合后续高级配置。

## 22. 归一化讨论

用户担心：

> 如果贡献系数翻倍，会不会超过最初设定的正常阶级分发金额？因为每 100 只有 15。

讨论确认：

贡献系数大于 1 时，确实可能导致总额超过返利池。

示例：

```text
第2级原本拿 3.60
贡献系数 1.2
第2级理想金额 = 4.32
总额可能超过 15
```

提出两种方案：

### 22.1 平台额外承担

- 活跃用户加成部分由平台额外支付。
- 激励更强。
- 但成本不可控。

### 22.2 归一化

- 先计算所有人的理想金额。
- 如果总额超过返利池，就按比例缩放。
- 总金额永远等于返利池。

用户明确选择：

> 我选择 A 归一。

也就是：

```text
最终总返利永远控制在返利池内。
```

归一化公式：

```text
缩放因子 = 返利池 / 理想总额
实际金额 = 理想金额 * 缩放因子
```

## 23. 贡献系数与防躺平

用户问：

> 防躺平机制是什么？

解释后，用户继续提出：

> 防躺平机制需要有惩有奖。  
> 又不想让他躺平，又不想让人家白白付出劳动。

因此防躺平机制的设计目标：

- 防止高层级用户长期不活跃却持续拿返利。
- 保护早期真实付出的人。
- 有奖励，也有惩罚。
- 可恢复。
- 可配置。

推荐机制：

```text
贡献系数 = 1.0 + 奖励加成 - 惩罚扣减
```

贡献系数范围示例：

```text
最小 0.3
最大 1.5
```

奖励条件示例：

```text
最近30天自充 >= 100 -> +0.1
最近30天自充 >= 200 -> +0.2
最近30天自充 >= 500 -> +0.3

最近30天新增有效下级 >= 1 -> +0.1
最近30天新增有效下级 >= 2 -> +0.2
最近30天新增有效下级 >= 3 -> +0.3
```

惩罚条件示例：

```text
连续30天无贡献 -> 系数 0.7
连续60天无贡献 -> 系数 0.5
连续90天无贡献 -> 系数 0.3
```

缓冲与恢复：

- 30 天宽限期。
- 重新活跃后可恢复。
- 历史贡献高的用户可保护最低系数。

## 25. 贡献系数与衰减系数是否冲突

用户问：

> 这个机制和刚刚说的衰减系数会不会冲突？

结论：

不冲突。

```text

```

## 36. 方案研讨总结

这次研讨的核心价值在于把问题从一个“返利触发缺陷”升级成了完整商业系统设计。

最重要的几次纠正是：

1. 不要直接依赖改 Sub2API 源码。
2. 不要把返利余额和 Sub2API 额度混在一起。
3. 不要误解成充值 100 返 100/200。
4. 正确规则是默认充值 100，拿 15 做返利池。
5. 首次拉人奖励不是单次充值，而是新人累计充值里程碑。
6. 里程碑奖励只给直接上级。
7. 里程碑结束后才进入正常多级分发。
8. 贡献系数会导致超额，所以必须归一化。
9. 防躺平要有惩有奖，还要保护历史贡献。
10. 所有配置要有 Tips，所有资金变化要有流水。

最终方向：

> Sub2Rebate 应该先做一个独立、可配置、可审计的返利闭环。  
> 第一版重点是邀请树、里程碑奖励、多级分发、独立余额和人工提现。  
> 等核心业务跑通后，再逐步加入支付宝自动化、贡献系数、防躺平、活动配置和高级分发算法。

## 37. 2026-06-17 邀请树失效节点与返利层级补充决策

本节用于补充 Sub2API 用户删除、系统判定用户不活跃、以及后续防躺平机制生效时的邀请树和返利处理规则。

### 37.1 核心原则

邀请树表示历史关系，不因为中间节点失效而自动重排。

返利资格表示当前是否可以继续获得新返利，和账号能否登录分开处理。

因此：

```text
账号能否登录
≠ 节点是否在邀请树上
≠ 节点是否能获得返利
```

### 37.2 中间节点失效后的树结构

示例：

```text
A
└── B
    └── C
        └── D
```

当 B 被 Sub2API 删除，或被系统防躺平/不活跃机制判定为返利失效时：

- C 不上浮到 A 名下。
- C 及其以下节点仍保留在 B 的下级。
- `referral_paths.path` 继续保留历史链路，例如 `A/B/C/D`。
- 管理端邀请树中 B 置灰，显示为已失效或返利失效。
- 用户端统计默认按有效节点展示，管理端需要同时展示结构总数、有效数和失效数。

也就是说，树仍然是：

```text
A
└── B（失效/置灰）
    └── C
        └── D
```

不能自动变成：

```text
A
└── C
    └── D
```

### 37.3 失效节点登录与返利资格

失效节点可以继续登录，除非账号状态被明确封禁。

建议将账号状态和返利资格拆开：

| 维度 | 建议字段 | 作用 |
|---|---|---|
| 账号状态 | `users.status` | 控制能否登录，例如 `active / banned` |
| 返利资格 | `users.rebate_status` | 控制能否获得新返利，例如 `eligible / disabled` |
| 树节点展示 | 可由 `rebate_status` 或节点状态派生 | 控制管理端树节点是否置灰 |

这样 B 可以是：

```text
status = active
rebate_status = disabled
```

含义是：B 可以登录查看自己的账号和历史数据，但不再获得任何新返利。

### 37.4 里程碑奖励口径

里程碑奖励不参与多级层级返利。

当前历史研讨中存在“新人累计充值达到里程碑后奖励直接上级”的描述，后续需要在实现前进一步收口最终口径。

本次补充决策强调：

- 里程碑是独立阶段，不参与多级衰减返利。
- 里程碑计算不要因为中间节点失效而触发上浮。
- 如果仍保留“奖励直接上级”的旧口径，则直接上级返利资格失效时，该笔里程碑奖励不自动转给更上级。
- 如果后续改成“基于个人的里程碑权益”，则里程碑只和充值用户自己的累计充值有关，不走邀请层级。

### 37.5 正常多级返利中失效节点的处理

正常多级返利读取完整历史链路，但遇到失效节点时，不给该节点发放新返利。

失效节点对应金额如何处理，由后台配置决定。

建议新增配置：

```text
rebate.inactive_node_mode
```

可选值：

| 选项 | 含义 | 适合场景 |
|---|---|---|
| `platform` | 按原始层级计算，失效节点那一份不发，归平台保留 | 默认推荐，关系和层级解释最稳定 |
| `exclude_recalculate` | 计算前排除失效节点，对剩余有效上级按衰减系数重新归一化 | 更偏激励有效上级，但会让上级因中间人失效而收益变多 |

示例：

```text
C -> B（失效）-> A
```

模式一：归平台

```text
第 1 级 B：不发，归平台
第 2 级 A：仍按第 2 级权重获得返利
```

模式二：排除失效节点后重算

```text
排除 B
A 成为本次计算中的第 1 个有效上级
按剩余有效上级重新计算衰减和归一化
```

默认建议使用 `platform`，避免因为某个节点失效导致其上级收益突然变高；后续可由管理员在配置中心切换。

### 37.6 配置中心 Tooltip 文案

配置中心新增“失效节点返利处理方式”，鼠标悬浮提示：

归平台：

```text
失效节点对应的返利不再发放，也不会转给其他上级，该部分金额归平台保留。树结构和原始层级不变。
```

排除失效节点后重算：

```text
计算层级返利时会跳过失效节点，并按剩余有效上级重新计算衰减权重。上级可能获得比原始层级更高的返利。
```

### 37.7 当前防躺平机制状态

当前代码层面尚未真正启用“防躺平贡献系数”。

现状是：

- 文档中已有贡献系数、防躺平、历史贡献保护的设计。
- 配置中心当前已有基础风控配置，例如黑名单和防重复发放检查。
- 返利计算当前主要按衰减系数和归一化分配。
- 当前没有自动根据活跃度把用户判定为返利失效。
- 当前没有贡献系数参与最终返利金额计算。

因此后续防躺平应作为独立实施项：

```text
先记录活跃/贡献数据
-> 再计算 rebate_status 或 contribution_factor
-> 最后接入返利计算
```
