# 2026-06-16 Sub2API 删号与邀请树问题清单

## 1. 范围说明

本文讨论的是 **Sub2API 用户账号** 删除场景，也就是共享用户体系里的 `users` / `user_affiliates`。

不讨论 `admin/accounts` 上游模型账号监控，因为那套数据当前只用于运营看板，不参与邀请树和返利计算。

## 2. 先回答两个核心问题

### 2.1 如果 Sub2API 账号删除，节点数怎么办？

建议不要再把“节点数”当成一个数字，而是拆成 3 套口径：

| 口径 | 含义 | 建议用途 |
|---|---|---|
| 结构节点数 | 邀请树里历史存在过的全部节点，包含已删号 / tombstone 节点 | 审计、树关系复盘、历史对账 |
| 有效节点数 | 当前仍可正常使用系统的节点，排除 `upstream_deleted`、`banned`、`tombstone` | 用户端展示、运营 KPI、转化率 |
| 可返利上级数 | 当前返利计算时真正可参与分发的上级人数 | 只给返利引擎内部使用 |

默认建议：

- 用户端展示 `有效节点数`
- 管理端同时展示 `结构节点数 / 有效节点数 / 已删号节点数`
- 返利计算只认 `可返利上级数`

### 2.2 如果 Sub2API 直接删除账号，本地树怎么办？

建议是：

1. **本地节点保留**
2. 标记为 `upstream_deleted` 或 tombstone
3. 撤销该用户 token，禁止继续登录
4. 保留 `referral_paths` 关系，不要把整条链路删掉
5. 新返利计算时按“可返利上级过滤规则”决定是否跳过该节点

原因很简单：

- Sub2API 账号是上游身份源
- 本地 `users + referral_paths + rebate_records + payment_records` 是你自己的业务账本和关系索引

上游删号只代表“这个人以后不能继续当活跃用户”，**不代表历史关系和历史账务应该被物理抹掉**。

## 3. 结合现有代码的结论

### 3.1 当前邀请树事实源已经是本地，不是 Sub2API 运行时直接树

关键代码：

- `backend/app/Modules/Invite/Services/InviteService.php`
- `backend/database/migrations/2026_06_13_000004_create_referral_paths_table.php`

当前系统是：

- 登录 / 资料刷新时，从 Sub2API 读取 `inviter_id`
- 再把关系写入本地 `referral_paths`
- 里程碑、衰减返利、推广统计都读取本地 `referral_paths.path`

所以你现在已经不是“完全跟着 Sub2API 活”。

**真正参与业务的是本地树。**

### 3.2 现在最大的危险不是“删号不同步”，而是“本地一旦硬删会把树和账一起打碎”

当前表设计里有两个危险组合：

1. `referral_paths.parent_user_id` 使用 `nullOnDelete`
2. `payment_records`、`rebate_events`、`rebate_records`、`rebate_balances`、`user_rebate_progress` 对 `users` 大量使用 `cascadeOnDelete`

这意味着：

- 删父节点，子节点不会自动重算 `path/depth`
- 删用户，历史充值、返利、余额、里程碑进度会被级联清掉

这不是普通脏数据，是**业务历史直接消失**。

## 4. 问题清单 + 修复方案 + 影响面

### P0-1：本地硬删用户会导致邀请树断裂

#### 现象

- 子节点 `parent_user_id` 会变成 `null`
- 但 `path` 和 `depth` 不会自动重算
- 管理端树会把子节点当成新的 root
- 团队统计仍可能沿旧 `path` 计算

#### 现有代码位置

- `backend/database/migrations/2026_06_13_000004_create_referral_paths_table.php`
- `backend/app/Modules/Invite/Services/InviteService.php`
- `backend/app/Http/Controllers/Api/V1/Admin/AdminRelationshipController.php`
- `backend/app/Http/Controllers/Api/V1/PromotionController.php`
- `backend/app/Http/Controllers/Api/V1/DashboardController.php`

#### 影响面

- 邀请树展示错乱
- 直邀 / 团队人数口径不一致
- 转化率分母漂移
- 后续返利链路和实际树结构不一致

#### 修复建议

- 禁止本地硬删 `users`
- 把用户生命周期改成 `active / banned / upstream_deleted / tombstone`
- 保留 `referral_paths`，不要让 `parent_user_id` 自动变 `null`
- 后续如果要做树迁移，必须走显式“迁移子树 + 重算 path/depth”的管理动作

### P0-2：本地硬删用户会级联删除历史账务

#### 现象

当前这些表都跟 `users` 有级联删除关系：

- `payment_records`
- `rebate_events`
- `rebate_records`
- `rebate_balances`
- `user_rebate_progress`
- 以及部分提现 / 风控数据

#### 影响面

- 历史充值记录消失
- 历史返利流水消失
- 已发过的钱追不回来
- 对账、审计、财务复盘失效

#### 修复建议

- 业务层禁止删除本地用户
- 数据库层建议逐步把关键历史表的 `cascadeOnDelete` 改成 `restrictOnDelete` 或至少保证应用层永远不执行删除
- 如果担心隐私，可以做“字段脱敏”，不要做“行删除”

### P0-3：上游删号目前没有正式对账 / 回收流程

#### 现象

当前只有登录和资料页会顺手同步用户快照：

- `backend/app/Modules/Auth/Services/Sub2RebateAuthService.php`
- `backend/app/Modules/User/Services/AccountProfileService.php`
- `backend/app/Modules/Invite/Services/InviteService.php`

如果上游账号被删，而该用户近期没有登录 / 没有人主动打开资料页，本地可能会长期保留 `active` 状态。

#### 影响面

- 已删号用户仍保留活跃 token
- 管理端 / 用户端仍把该节点当有效成员
- 新返利可能继续发给本地残留的“幽灵用户”

#### 修复建议

- 增加 **上游用户生命周期对账任务**
- 不是登录时才同步，而是定时扫本地活跃用户与上游用户列表
- 建议不要“缺失一次就判死”，而是：
  - 第 1 次缺失：记录 `sync_missing`
  - 连续 N 次缺失：标记 `upstream_deleted`
  - 恢复出现：自动恢复 `active`

这样能防止把临时网络故障误判成删号。

### P0-4：返利计算还没有“可返利上级过滤”

#### 现象

当前：

- `MilestoneService` 直接拿 `ancestorIds()[0]` 当直接上级
- `DecayRebateService` 直接拿 `ancestorIds()` 全链路发衰减返利

而 `ancestorIds()` 只按 `path` 拆 ID，不校验这些上级是否仍然可用。

#### 影响面

- 如果本地残留节点仍是 `active`，删号用户可能继续拿新返利
- 如果以后真的硬删了本地用户，返利插入又可能因为外键失败直接报错

#### 修复建议

把“邀请结构链路”和“返利可用链路”拆开：

- `ancestorIds()` 继续保留，用于结构复盘
- 新增 `eligibleAncestorIds()`，只返回当前可发钱的上级

建议的业务规则：

- **里程碑奖励**：默认只给直接邀请人；直接邀请人若已 `upstream_deleted`，建议默认不自动转给更上级，等待管理员迁移或人工处理
- **衰减返利**：可以跳过已删号 / 已封禁节点，对剩余可用上级重新归一化分配

这样最符合你现有的“里程碑奖励只给直推，多级返利用衰减系数分池”的业务语义。

### P1-5：节点数和统计口径现在是混的

#### 现象

当前接口基本只有：

- `directInviteCount`
- `teamInviteCount`

但这两个值没有区分：

- 是否包含已删号节点
- 是否包含仅保留结构的 tombstone 节点
- 是否是返利可用链路里的有效人数

#### 修复建议

建议把统计字段拆开：

- `directInviteCount`: 有效直邀数
- `directInviteTotalCount`: 结构直邀总数
- `directDeletedCount`: 已删号直邀数
- `teamInviteCount`: 有效团队数
- `teamInviteTotalCount`: 结构团队总数
- `teamDeletedCount`: 已删号团队数

用户端默认展示有效数，管理端同时展示总数和删号数。

### P1-6：删除 / tombstone / 孤儿链路没有测试覆盖

#### 现象

当前深度测试覆盖了：

- 邀请绑定
- Sub2API `inviter_id` 同步
- 返利计算
- 幂等

但没有专门覆盖：

- 上游用户删除
- 本地用户 tombstone
- 直接上级失效后的里程碑策略
- 中间层失效后的衰减返利重分配
- 树显示与统计字段双口径

#### 修复建议

至少补这几类测试：

1. A -> B -> C，B `upstream_deleted` 后，树结构仍保留，统计分有效 / 结构两套口径
2. B `upstream_deleted` 后，C 的里程碑奖励不再自动给 B
3. B `upstream_deleted` 后，C 的衰减返利跳过 B，重新归一给 A
4. 已删号用户已有历史 `rebate_records` 和 `payment_records`，后续同步不会丢历史账

## 5. 推荐的业务策略

### 5.1 节点保留策略

- 本地 `users` 永不硬删
- 上游删号后转 `upstream_deleted`
- 允许脱敏显示，但不删结构行和账务行

### 5.2 统计策略

- 用户端：看有效节点数
- 管理端：看有效数 + 结构总数 + 已删号数
- 返利引擎：只看可返利上级数

### 5.3 返利策略

#### 里程碑奖励

建议规则：

- 仍然坚持“只给直接邀请人”
- 直接邀请人删号后，不自动改发给爷爷节点
- 若业务想恢复该支线收益，必须由管理员手工迁移子树

这样不会悄悄改变“直推奖励”的业务含义。

#### 衰减返利

建议规则：

- 跳过 `upstream_deleted / tombstone / banned` 节点
- 对剩余可用上级重新归一化分完整个返利池

这样既保留了无限级金字塔逻辑，也不会把钱发给失效节点。

## 6. 推荐实施顺序

### 第一阶段：先止血

1. 禁止本地硬删用户
2. 增加 `upstream_deleted` / tombstone 生命周期
3. 对已删号用户撤销 token
4. 补上上游删号定时对账

### 第二阶段：修返利和统计

1. 新增 `eligibleAncestorIds()`
2. 里程碑 / 衰减返利分开处理失效上级
3. 统计字段拆成“有效 / 结构 / 已删号”三套口径

### 第三阶段：补管理动作和测试

1. 后台增加子树迁移 / 修正入口
2. 邀请树节点加 `upstream_deleted` 状态标识
3. 补全删号 / tombstone / 返利跳过 / 统计双口径测试

## 7. 测试验证计划

这一块建议你不要等“全做完再测”，而是每一阶段改完就验。

### 7.1 测试目标

这次测试要验证 4 件事：

1. 上游删号后，本地树结构不会丢
2. 节点统计会按“结构 / 有效 / 可返利”分口径正确计算
3. 返利不会继续发给失效节点
4. 历史充值、返利、余额、审计不会因为删号而消失

### 7.2 自动化测试计划

建议新增一个专项测试文件，比如：

- `backend/tests/Feature/InviteDeletionLifecycleTest.php`

如果你想少建文件，也可以先并入：

但从长期维护看，单独建专项文件更清楚。

#### A. 邀请树生命周期测试

用例 1：`A -> B -> C`，B 被标记 `upstream_deleted`

验证点：

- `users` 里 B 仍存在
- `referral_paths` 里 B、C 仍存在
- C 的 `parent_user_id` 仍然是 B
- C 的 `path` 仍然保留 `A/B/C` 链路
- 管理端树仍能看到 B 和 C
- B 节点状态显示为 `upstream_deleted`

用例 2：上游账号暂时查不到，第一次只记 `sync_missing`

验证点：

- 本地不立即改成 `upstream_deleted`
- 不会误踢正常用户

用例 3：连续 N 次缺失后再标记 `upstream_deleted`

- 到达阈值前不判死

- token 被撤销

#### B. 节点统计口径测试

用例 4：`A -> B -> C`，B 为 `upstream_deleted`

验证点：

- `teamInviteTotalCount = 2`
- `teamDeletedCount = 1`

这里建议你先定规则：

- 如果“有效团队成员”只看节点自身是否可用，那么 C 仍算有效
- 如果“有效团队成员”要求整条祖先链都有效，那 C 不算

我更建议第一种，口径更直观。

#### C. 返利引擎测试

验证点：

- 默认也不自动转给 A
- `rebate_records` 不新增 milestone 记录
- 事件状态和错误信息符合你的业务定义

用例 6：衰减返利，`A -> B -> C`，B `upstream_deleted`

验证点：

- `eligibleAncestorIds()` 只返回 A
- 返利池重新归一化后全部给 A
- 不会给 B 产生新的 decay 记录

用例 7：已删号节点历史记录保留

验证点：

- B 在删号前已有 `rebate_records`
- 标记 `upstream_deleted` 后历史记录仍存在

### 7.3 手工联调计划

自动化测的是逻辑，手工联调要测真实数据路径。

#### 联调场景 1：真实上游删号同步

步骤：

1. 在测试环境准备 `A -> B -> C`
2. 让 B 先有正常登录记录和邀请关系
3. 在 Sub2API 删除 B
4. 运行上游生命周期对账任务
5. 检查本地状态

预期：

- B 变成 `upstream_deleted`
- B token 失效
- B/C 邀请链仍可在管理端看到
- 历史返利和充值记录都还在

#### 联调场景 2：删号后充值返利

步骤：

1. 保持 `A -> B -> C`
2. 删除上游 B
3. 给 C 造一笔充值事件
4. 跑返利任务

预期：

- 里程碑奖励不再发给 B
- 衰减返利按新规则跳过 B
- 不出现外键错误
- 不出现重复发放

#### 联调场景 3：统计页验证

步骤：

1. 打开用户端推广页
2. 打开管理端关系树
3. 对比数据库统计

预期：

- 用户端展示有效数
- 管理端能看到总数、有效数、删号数
- 页面口径和 SQL 对得上

## 8. 最终建议

对你这个项目，最稳的原则只有一句话：

**Sub2API 账号可以删，但本地业务节点不能跟着物理删。**

上游删号解决的是“这个人以后还能不能继续用系统”，本地树解决的是“这条关系和这笔账以前是否存在过”。

这两件事必须拆开处理，不然邀请树、节点统计、返利流水、余额和审计都会串。
