# 2026-06-17 优化返利层级方案实施计划

## 1. 背景

当前 Sub2Rebate 的邀请树已经落在本地 `users` / `referral_paths` 中。Sub2API 只提供用户和直接邀请人关系，本地负责沉淀树结构、统计和返利链路。

当中间节点被 Sub2API 删除、失联，或后续被防躺平机制判定为不活跃时，不能直接删除本地节点，也不能把下级自动上浮到更上级名下。

本次优化目标是：

```text
树结构保持原样
失效节点置灰
失效节点可登录
失效节点不再获得新返利
失效节点对应返利按配置处理
轻量防躺平：连续 7 天无活跃行为则置灰并停止参与返利，有充值则恢复
```

## 2. 最终业务规则

示例：

```text
A
└── B
    └── C
        └── D
```

当 B 失效：

- C 不挂到 A 名下。
- C 及以下节点仍保留在 B 的下级。
- B 在管理端邀请树中置灰。
- B 可以登录查看账号和历史数据。
- B 不获得任何新的里程碑或多级返利。
- B 如果连续 7 天账户余额无充值、无减少，且没有发展新下级，则由防躺平机制置灰并暂停返利资格。
- B 后续只要出现充值行为，则恢复返利资格。
- 历史 `path`、历史流水、历史账务不重算、不删除。
- 3 级、5 级、无限级邀请树都按同一规则处理。

## 3. 字段与配置计划

### 3.1 用户返利资格字段

建议新增字段：

```text
users.rebate_status
```

可选值：

| 值 | 含义 |
|---|---|
| `eligible` | 可获得新返利 |
| `disabled` | 不可获得新返利，但默认仍可登录 |

保留现有：

```text
users.status
```

用于控制账号登录，例如：

| 值 | 含义 |
|---|---|
| `active` | 可登录 |
| `banned` | 禁止登录 |

### 3.2 失效节点返利处理配置

新增配置：

```text
rebate.inactive_node_mode
```

默认建议：

```text
platform
```

可选值：

| 值 | 名称 | 说明 |
|---|---|---|
| `platform` | 归平台 | 按原始层级计算，失效节点那份不发，归平台保留 |
| `exclude_recalculate` | 排除后重算 | 计算前排除失效节点，对剩余有效上级重新按衰减系数归一化 |

### 3.3 轻量防躺平配置

新增配置：

```text
risk.lie_flat_enabled = true
risk.lie_flat_days = 7
```

判断规则：

```text
连续 7 天没有充值
连续 7 天账户余额没有减少
连续 7 天没有发展新下级
=> rebate_status = disabled
```

恢复规则：

```text
用户出现充值行为
=> rebate_status = eligible
```

说明：

- 账户余额变化只用于判断用户是否活跃。
- 账户余额变化不能作为返利事件来源。
- 防躺平只改变返利资格，不改邀请树，不删除节点，不迁移下级。
- 当前 Sub2API 用户读取层已经可以读取 `users.balance` 和 `users.total_recharged`，可用于本机制的活跃监控。

## 4. 后端实施步骤

### 阶段 A：数据结构

1. 新增迁移，给 `users` 增加 `rebate_status` 字段。
2. 默认值为 `eligible`。
3. 增加索引，方便统计和返利过滤。
4. 默认配置中加入 `rebate.inactive_node_mode`。
5. 默认配置中加入 `risk.lie_flat_enabled` 和 `risk.lie_flat_days`。
6. 建议增加轻量活跃字段，避免每次全表回溯计算：
   - `users.last_recharge_at`
   - `users.last_balance_decreased_at`
   - `users.last_invited_at`
   - `users.rebate_disabled_at`
   - `users.rebate_disabled_reason`
   - `users.last_sub2api_balance`
   - `users.last_sub2api_total_recharged`
   - `users.last_balance_checked_at`
7. 如需保留快照历史，可新增 `sub2api_user_balance_snapshots`：
   - `user_id`
   - `balance`
   - `total_recharged`
   - `checked_at`
8. 配置接口返回这些配置项和 tips。

### 阶段 B：邀请树

1. `InviteService::ancestorIds()` 保持结构链路，不过滤失效节点。
2. 新增返利用链路方法，返回上级 ID 和原始层级。
3. 邀请树节点返回 `rebateStatus`。
4. 管理端关系树接口返回失效节点状态，供前端置灰。
5. 不做任何自动上浮、自动迁移、自动重写 `path`。

### 阶段 C：返利计算

1. 里程碑奖励不参与多级层级返利。
2. 若当前仍按“直接上级获得里程碑”实现，则直接上级 `rebate_status = disabled` 时不发，也不转给更上级。
3. 衰减返利读取完整结构链路。
4. 如果配置为 `platform`：
   - 保留原始层级。
   - 失效节点不写 `rebate_records`。
   - 失效节点对应金额归平台。
   - 有效节点按原始层级权重获得返利。
5. 如果配置为 `exclude_recalculate`：
   - 先排除失效节点。
   - 剩余有效上级重新编号。
   - 按衰减系数重新归一化分配。

### 阶段 D：后台与审计

1. 用户详情展示 `rebate_status`。
2. 用户列表支持筛选返利资格。
3. 管理端关系树中返利失效节点置灰。
4. 配置中心新增“失效节点返利处理方式”。
5. 配置中心新增“防躺平开关”和“连续无活跃天数”。
6. 所有返利资格变更写审计日志。

### 阶段 E：轻量防躺平任务

1. 新增定时任务，例如 `rebate:check-lie-flat-users`。
2. 定时扫描 `status = active` 且 `rebate_status = eligible` 的用户。
3. 通过 `Sub2ApiUserRepository::findById($id)` 读取 Sub2API 用户余额：
   - `balance`
   - `totalRecharged`
4. 和本地上一次记录对比：
   - `balance` 变小：说明用户仍在使用服务，更新 `last_balance_decreased_at`。
   - `totalRecharged` 变大：说明用户有充值迹象，更新 `last_recharge_at`，并可恢复返利资格。
   - `balance` 变大：只记录快照，不自动触发返利。
5. 判断最近活跃时间：
   - 最近充值时间。
   - 最近余额减少时间。
   - 最近发展新下级时间。
6. 如果三类行为都连续 N 天没有发生，则改为：
   - `rebate_status = disabled`
   - `rebate_disabled_at = now()`
   - `rebate_disabled_reason = lie_flat`
7. 用户发生充值时恢复：
   - `rebate_status = eligible`
   - 清空或更新失效原因。
8. 邀请树结构保持原样，只影响置灰和返利资格。

注意：

- `balance` 减少可以作为活跃使用信号。
- `totalRecharged` 增加可以作为充值恢复信号。
- `balance` 增加不能直接作为自动返利事件来源。
- 真正发返利仍然必须走明确充值事件，例如 Sub2Rebate 充值订单、Sub2API `payment_orders`、`redeem_codes`、`payment_audit_logs` 或后台补录事件。

## 5. 前端实施步骤

### 配置中心

新增选项：

```text
失效节点返利处理方式
```

选项：

- 归平台
- 排除失效节点后重算

Tooltip：

```text
归平台：失效节点对应的返利不再发放，也不会转给其他上级，该部分金额归平台保留。树结构和原始层级不变。
```

```text
排除失效节点后重算：计算层级返利时会跳过失效节点，并按剩余有效上级重新计算衰减权重。上级可能获得比原始层级更高的返利。
```

### 邀请树

1. 管理端节点根据 `rebateStatus = disabled` 置灰。
2. 节点详情展示“返利资格：已失效 / 正常”。
3. 节点详情展示失效原因，例如“防躺平：连续 7 天无活跃”。
4. 用户端默认展示有效统计，避免把失效节点当作活跃贡献。

### 防躺平配置

新增选项：

- 启用防躺平检测。
- 连续无活跃天数，默认 7 天。

Tooltip：

```text
开启后，系统会监控用户是否有充值、账户余额减少或发展新下级。连续指定天数都没有这些活跃行为时，该用户会被置灰并暂停获得新返利；后续有充值会自动恢复。
```

```text
账户余额变化只用于判断活跃度，不作为自动发放返利的依据。
```

## 6. 测试计划

建议新增：

```text
backend/tests/Feature/RebateInactiveNodeTest.php
```

重点用例：

1. A -> B -> C，B 返利失效后，C 仍在 B 名下。
2. B 返利失效后，B 可以登录。
3. B 返利失效后，C 充值不再给 B 产生新返利。
4. `platform` 模式下，B 那份金额不发，A 保持原始层级权重。
5. `exclude_recalculate` 模式下，排除 B 后 A 重新按有效上级链路计算。
6. B 恢复 `eligible` 后，后续新事件可以继续获得返利，历史事件不重算。
7. 管理端关系树返回 B 的失效状态。
8. 用户连续 7 天无充值、余额无减少、无新增下级后，防躺平任务将其 `rebate_status` 改为 `disabled`。
9. 被防躺平置灰的用户出现充值后，自动恢复为 `eligible`。
10. 余额变化只影响活跃判断，不会单独创建返利事件。
11. Sub2API `balance` 下降会更新 `last_balance_decreased_at`，避免正常使用中的用户被误判躺平。
12. Sub2API `balance` 上升不会创建 `rebate_events`，防止把管理员调额、退款等误当充值返利。

## 7. 验收标准

- 邀请树结构不会因为中间节点失效而变化。
- 失效节点可登录，但不能获得新返利。
- 管理端能清楚看到置灰节点。
- 配置中心可以切换失效节点返利处理方式。
- 配置中心可以开启/关闭轻量防躺平，并配置连续无活跃天数。
- 连续 7 天无充值、余额无减少、无新增下级的用户会置灰并暂停新返利。
- 用户充值后会自动恢复返利资格。
- 两种处理方式都有测试覆盖。
- 旧的正常返利、里程碑、提现、审计测试不回退。

## 8. 本次纳入的轻量防躺平机制

本次不做复杂贡献系数，不做历史贡献保护，也不调整返利金额权重。

本次只做轻量版：

```text
监控活跃行为
-> 连续 N 天无活跃
-> 置灰用户
-> 暂停参与新返利
-> 有充值后恢复
```

活跃行为包括：

- 有充值。
- 账户余额有减少，说明仍在使用服务。
- 有发展新下级。

不活跃判断：

```text
默认连续 7 天没有充值、没有余额减少、没有发展新下级
=> 返利资格 disabled
=> 管理端置灰
=> 不参与新返利
```

恢复判断：

```text
用户有充值
=> 返利资格 eligible
=> 取消置灰
=> 后续新事件可参与返利
```

注意：

- 防躺平不改变 `referral_paths`。
- 防躺平不删除用户。
- 防躺平不影响登录。
- 防躺平不重算历史返利。
- 账户余额监控只用于活跃判断，不用于自动触发返利。

## 9. Sub2API 余额监控实现细节

当前代码已具备读取 Sub2API 用户余额的基础：

```text
Sub2ApiUserRepository::findById($id)
-> Sub2ApiUserData.balance
-> Sub2ApiUserData.totalRecharged
```

读取来源：

```text
Sub2API users.balance
Sub2API users.total_recharged
```

建议流程：

```text
定时任务
-> 扫描本地用户
-> 按 ID 查询 Sub2API 用户
-> 读取 balance / total_recharged
-> 与本地 last_sub2api_balance / last_sub2api_total_recharged 对比
-> 更新 last_balance_decreased_at / last_recharge_at
-> 写入最新快照
-> 再执行 7 天无活跃判断
```

判断规则：

```text
balance < last_sub2api_balance
=> 用户余额减少
=> 视为仍在使用服务
=> 更新 last_balance_decreased_at
```

```text
total_recharged > last_sub2api_total_recharged
=> 用户存在充值增长
=> 更新 last_recharge_at
=> 如果 rebate_status = disabled，可恢复 eligible
```

```text
balance > last_sub2api_balance
=> 只更新余额快照
=> 不自动创建返利事件
```

原因：

- 余额减少通常说明用户在调用服务，有活跃行为。
- 总充值增加可以作为恢复返利资格的信号。
- 余额增加来源复杂，可能是充值、管理员调额、退款或其他内部操作，不能直接用于自动发返利。

最终边界：

```text
余额监控负责判断活跃
充值事件负责触发返利
两者不能混在一起
```
